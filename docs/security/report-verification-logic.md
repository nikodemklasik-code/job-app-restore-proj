# Report Verification Logic
**Status:** Production Security Specification  
**Priority:** P0  
**Product:** English-first Real Conversational Interviewer / Coaching Reports  
**Audience:** Engineering, Security, Product, QA  
**Owner:** Engineering / Security  
**Last Updated:** 2026-04-10

---

## 1. Purpose

This document defines the production verification logic for system-generated coaching and interview reports used for cross-session comparison.

Its purpose is to ensure that a prior-session PDF can be trusted **only if**:
- it was generated by the system,
- its metadata is internally consistent,
- its verification code matches canonical report data,
- its comparison eligibility is explicitly enabled.

This document is the source of truth for:
- how `VCODE` is generated,
- what enters the canonical verification payload,
- how signing and versioning work,
- how backend validation must be performed.

---

## 2. Security Objective

The verification system must prevent three classes of failure:

1. **Tampered report acceptance**  
   A user edits report values manually and still gets comparison access.

2. **Format drift**  
   Different code paths generate different verification outputs for logically identical reports.

3. **Silent weakening of integrity logic**  
   Future implementation changes reduce verification strictness without explicit approval.

The validation system must therefore be:
- deterministic,
- canonical,
- versioned,
- fail-closed,
- easy to audit.

---

## 3. Core Security Rule

A prior report is valid for comparison only if all of the following are true:
- all required report fields are present,
- all required validation block fields are present,
- metadata is structurally valid,
- metadata is logically consistent,
- the document signature format is valid,
- `Comparison Eligibility` is `YES`,
- `VCODE` matches the canonical verification payload for the document.

If any of the above fails, the report must be treated as invalid for historical comparison.

---

## 4. Verification Components

The verification mechanism has four core components:

### 4.1 Report Metadata
Top-level report fields embedded in the generated PDF.

### 4.2 Validation Block
A strict field block used for internal verification.

### 4.3 Verification Code (`VCODE`)
A deterministic integrity code derived from canonical report data.

### 4.4 Document Signature
A report-level authenticity marker bound to the same report identity model, optionally stronger than `VCODE`.

`VCODE` is the minimum required integrity check.  
`Document Signature` may be used as an additional trust layer.

---

## 5. Required Fields

### 5.1 Required top-level fields
Every comparison-eligible report must contain:
- `Coaching Report ID`
- `Session Date`
- `Evaluator Version`
- `Comparison Eligibility`
- `Document Signature`
- `Validation Block`

### 5.2 Required validation block fields
The `Validation Block` must contain:
- `RID`
- `VER`
- `ANSWERS`
- `AVG`
- `VCODE`

Missing any of the above must cause validation failure.

---

## 6. Canonical Verification Payload

### 6.1 Goal
The canonical verification payload is the exact normalized data used to generate and verify `VCODE`.

This payload must:
- have a fixed schema,
- have a fixed field order,
- have fixed normalization rules,
- be stable across environments,
- be version-controlled.

### 6.2 Minimum required fields in canonical payload
The following values must be included:

- `report_type`
- `session_type`
- `report_id`
- `session_date`
- `evaluator_version`
- `answers_count`
- `average_score`
- `comparison_eligibility`

### Required semantics
- `report_type` distinguishes report families if more than one exists
- `session_type` distinguishes Interview Coach, Negotiation Skill, etc.
- `report_id` uniquely identifies the report
- `session_date` reflects the report's source session date
- `evaluator_version` identifies evaluator logic version
- `answers_count` is the number of evaluated answers
- `average_score` is the normalized reported average
- `comparison_eligibility` must be included to prevent a no-compare report from being transformed into a compare-eligible one by field editing

### 6.3 Recommended optional fields
The following fields are recommended if stable enough:
- `strengths_hash`
- `gaps_hash`
- `summary_hash`
- `schema_version`

These improve tamper resistance, but should only be included if they can be normalized reliably.

---

## 7. Canonicalization Rules

Canonicalization must happen before hashing or signing.

### 7.1 Field ordering
Fields must always appear in the exact same order in the canonical payload:

1. `schema_version`
2. `report_type`
3. `session_type`
4. `report_id`
5. `session_date`
6. `evaluator_version`
7. `answers_count`
8. `average_score`
9. `comparison_eligibility`
10. optional hashed content fields

No alternative order is allowed.

### 7.2 Value normalization
The following normalization rules are required:

#### Strings
- trim leading/trailing whitespace
- collapse internal repeated whitespace to single spaces only if the field type allows it
- preserve case unless explicitly defined otherwise
- do not localize values

#### Dates
- must use ISO format: `YYYY-MM-DD`

#### Numbers
- `answers_count` must be serialized as base-10 integer string
- `average_score` must be normalized to fixed decimal precision

### 7.3 Average score normalization
`average_score` must be normalized to exactly one decimal place unless product policy changes.

Examples:
- `7.2` → `7.2`
- `7.20` → `7.2`
- `7.24` → `7.2` after chosen rounding strategy
- `7` → `7.0`

The rounding strategy must be explicitly chosen and fixed. Recommended:
- round half away from zero at one decimal place

### 7.4 Eligibility normalization
`comparison_eligibility` must serialize as:
- `YES`
- `NO`

No lowercase or boolean form is allowed in canonical payload.

### 7.5 Delimiter format
Canonical payload must use strict delimiter formatting.

Recommended canonical serialization:
```text
schema_version=RV1|report_type=COACHING_REPORT|session_type=INTERVIEW_COACH|report_id=CR-2026-04-11-84172|session_date=2026-04-11|evaluator_version=INT-v1.0|answers_count=6|average_score=7.2|comparison_eligibility=YES
```

This format must not vary between code paths.

## 8. VCODE Generation Logic

### 8.1 Purpose

`VCODE` is a deterministic integrity code derived from canonical report metadata and a server-side secret.

It is used to detect:

* metadata tampering,
* report field inconsistencies,
* fake system reports,
* manual edits to comparison-relevant values.

### 8.2 Required algorithm behavior

`VCODE` generation must:

* use canonical payload only,
* include a server-side secret or HMAC key,
* be deterministic,
* be version-aware.

### 8.3 Recommended implementation

Recommended production implementation:

* `HMAC-SHA256(secret, canonical_payload)`

This is preferable to raw hashing because it prevents offline forgery if the payload format is known.

### 8.4 Raw code generation example

Conceptual example:

```text
raw_vcode = HMAC_SHA256(report_verification_secret, canonical_payload)
```

### 8.5 Public display format

The stored/displayed `VCODE` may be:

* full encoded string, or
* shortened human-readable derived format.

Recommended approach:

* keep full HMAC internally for verification,
* expose a shortened display-safe representation if needed.

If only a shortened code is stored in the PDF, collision risk and truncation strategy must be explicitly documented.

### 8.6 Recommended display strategy

Recommended format:

* compute full HMAC hex or base64url
* derive a short formatted display code from a prefix or mapped fragment
* keep verification based on full recomputed comparison, not human formatting alone

Example:

```text
VCODE=VC-RV1-A13F9C72E4
```

The exact format is up to engineering, but it must include:

* a recognizable prefix
* verification logic version reference or derivable version mapping
* enough entropy to make trivial guessing impractical

## 9. Document Signature Logic

### 9.1 Purpose

`Document Signature` is a separate authenticity marker for the whole report identity.

It may be used:

* as a stronger artifact-level trust marker,
* as a debugging aid,
* as a support/reconciliation field.

### 9.2 Relationship to VCODE

`Document Signature` must not replace `VCODE`.

Recommended model:

* `VCODE` protects validation block / comparison-critical metadata
* `Document Signature` protects report identity and issue context

### 9.3 Recommended input set

`Document Signature` should include:

* canonical verification payload
* report generation timestamp
* generator/service identifier
* signing version

Example conceptual payload:

```text
signature_payload = canonical_payload + "|" + issued_at + "|" + generator_id + "|" + signing_version
```

### 9.4 Recommended implementation

Recommended production implementation:

* HMAC-SHA256 with separate secret  
  or
* asymmetric signature if the system later evolves into externally verifiable documents

At current stage, HMAC with a separate secret is acceptable and simpler.

## 10. Versioning Model

Versioning must be explicit and stable.

### 10.1 Required version fields

The system must maintain:

* `schema_version`
* `evaluator_version`
* `verification_logic_version`
* optionally `signature_version`

### 10.2 Meanings

* `schema_version`: report field format version
* `evaluator_version`: evaluation engine version
* `verification_logic_version`: canonicalization + integrity logic version
* `signature_version`: document signature generation version

### 10.3 Minimum rule

A verification request must be processed using the verification logic version appropriate for the report.

Example:

* old reports with `RV1` use `RV1` canonicalization and `RV1` verification rules
* new reports with `RV2` use `RV2` rules

No report should be validated using guessed version behavior.

## 11. Backend Validation Flow

The backend validation flow must be strict and ordered.

### 11.1 Step 1: Parse PDF

Extract:

* top-level fields
* validation block
* comparison eligibility field
* document signature
* report body values needed for consistency checks

If parsing fails:

* return `INVALID_PDF_STRUCTURE`

### 11.2 Step 2: Check required fields

Verify presence of all required fields.

If any are missing:

* return `INVALID_REQUIRED_FIELDS`

### 11.3 Step 3: Normalize extracted values

Normalize all fields according to canonicalization rules.

If normalization fails:

* return `INVALID_METADATA_CONSISTENCY`

### 11.4 Step 4: Validate field consistency

Check:

* `RID == Coaching Report ID`
* `VER == Evaluator Version`
* `ANSWERS == reported answers count`
* `AVG == normalized reported average`
* `Comparison Eligibility` is valid
* signature format is syntactically valid

If any mismatch occurs:

* return `INVALID_METADATA_CONSISTENCY`

### 11.5 Step 5: Select verification logic version

Determine:

* schema version
* verification logic version

If unsupported:

* return `UNSUPPORTED_VERIFICATION_VERSION`

### 11.6 Step 6: Rebuild canonical payload

Reconstruct canonical payload from normalized values only.

Do not trust the incoming `VCODE` as source input.

### 11.7 Step 7: Recompute expected VCODE

Use:

* correct versioned logic
* correct secret
* canonical payload

If recomputed code does not match reported `VCODE`:

* return `INVALID_VCODE`

### 11.8 Step 8: Validate comparison eligibility

If `Comparison Eligibility != YES`:

* return `COMPARISON_NOT_ALLOWED`

### 11.9 Step 9: Return final result

If everything passes:

* return `VALID_FOR_COMPARISON`

---

## 12. Validation Result Contract

Recommended backend result shape:

```json
{
  "status": "VALID_FOR_COMPARISON",
  "reportId": "CR-2026-04-11-84172",
  "verificationLogicVersion": "RV1",
  "comparisonEligibility": "YES",
  "failureReason": null
}
```

Recommended failure shape:

```json
{
  "status": "INVALID_VCODE",
  "reportId": "CR-2026-04-11-84172",
  "verificationLogicVersion": "RV1",
  "comparisonEligibility": "YES",
  "failureReason": "Computed verification code did not match report VCODE."
}
```

## 13. Fail-Closed Requirements

The system must fail closed in all ambiguous cases.

### Required fail-closed cases

* missing field
* malformed validation block
* unsupported version
* invalid normalization
* average mismatch
* answer count mismatch
* signature format mismatch
* invalid VCODE
* invalid eligibility value

### Explicit rule

No fuzzy matching, no best-effort trust, no partial acceptance.

If the report cannot be proven valid, it must be treated as invalid.

---

## 14. Secret Management

### 14.1 Required

Verification and signature secrets must:

* never be embedded in frontend code
* never be embedded in PDFs
* never be logged
* be stored in secure backend configuration

### 14.2 Recommended environment variables

Examples:

* `REPORT_VERIFICATION_SECRET_RV1`
* `REPORT_SIGNATURE_SECRET_SV1`

### 14.3 Secret rotation

Rotation must be supported by version-aware verification.

Old reports must remain verifiable through retained old secrets for as long as product policy allows.

---

## 15. Logging Rules

### Safe to log

* validation status
* report id
* schema version
* verification logic version
* comparison eligibility value
* failure category
* processing timestamp

### Must not log

* raw secrets
* canonical payload with secret
* raw HMAC output before formatting if unnecessary
* full report text by default
* full conversation content

---

## 16. QA and Tamper Test Cases

QA must test at minimum:

### Valid cases

* clean valid report
* valid report with comparison eligibility YES
* valid old-version report using old verification logic

### Invalid cases

* missing `Validation Block`
* missing `VCODE`
* edited `AVG`
* edited `ANSWERS`
* edited `RID`
* edited `Evaluator Version`
* changed `Comparison Eligibility`
* valid-looking but fake `VCODE`
* unsupported verification version
* malformed delimiter formatting
* invalid date normalization
* modified report content when content hashes are included

Each case must return the correct failure state.

---

## 17. Recommended Implementation Boundaries

Split implementation into the following backend units:

* `reportFieldExtractor`
* `reportNormalizer`
* `canonicalPayloadBuilder`
* `vcodeService`
* `documentSignatureService`
* `reportVerificationService`
* `verificationVersionRegistry`

This avoids the classic horror of one 400-line "verifyPdf.ts" file where every future bug gets buried with full honors.

## 18. Production Acceptance Criteria

This logic is production-ready only if:

* `VCODE` generation is deterministic
* canonical payload is versioned and documented
* validation is fail-closed
* secrets are server-side only
* version-aware verification works
* tampered reports fail reliably
* comparison is enabled only after full validation
* QA tamper tests pass
