# History Verification Policy
**Status:** Production Policy  
**Priority:** P0  
**Product:** English-first Real Conversational Interviewer / Coaching Reports  
**Audience:** Engineering, Product, QA, Security, Support  
**Owner:** Product / Engineering  
**Last Updated:** 2026-04-11

---

## 1. Purpose

This document defines the product policy for when and how prior-session coaching reports are accepted as valid input for cross-session comparison.

It is the policy-level companion to:
- `docs/security/report-verification-logic.md` (technical verification spec)

This document answers: **when** is a prior report eligible for comparison, and **what happens** when it is not.

---

## 2. Policy Scope

This policy applies to:
- all coaching reports generated by the system
- any feature that allows users to upload or submit a prior report for cross-session comparison
- any backend endpoint that processes uploaded report PDFs

---

## 3. Core Eligibility Rule

A prior-session report is eligible for comparison **only if** all of the following are true:

1. The report was generated by the system (not manually created or modified).
2. The report contains a valid Validation Block with all required fields.
3. The `Comparison Eligibility` field is explicitly set to `YES`.
4. The `VCODE` passes server-side verification.
5. All metadata consistency checks pass.

**No partial eligibility exists.** A report is either fully eligible or fully ineligible.

---

## 4. Comparison Eligibility Values

| Value | Meaning |
|-------|---------|
| `YES` | Report is eligible for cross-session comparison |
| `NO`  | Report is not eligible for comparison |

Any other value or absence of the field must be treated as `NO`.

---

## 5. When Comparison Eligibility is Set to YES

A report receives `Comparison Eligibility: YES` only when:
- the session was complete (not abandoned mid-session),
- the evaluation produced a valid score set,
- the report was generated by the production system,
- all required validation block fields were successfully populated.

Partial sessions, incomplete evaluations, or system-error reports must receive `Comparison Eligibility: NO`.

---

## 6. When Comparison Eligibility is Set to NO

A report must receive `Comparison Eligibility: NO` when any of the following apply:
- the session was abandoned or incomplete,
- the evaluation engine returned an error state,
- required fields could not be populated at generation time,
- the report was generated in a test or demo environment (unless explicitly overridden by policy),
- the product policy for the session type does not support comparison.

---

## 7. Uploading a Prior Report

When a user submits a prior report PDF for comparison:
1. The system must validate the report according to the technical verification spec.
2. If validation fails for any reason, the report must be rejected with a clear failure reason.
3. The user must be informed that the prior report could not be accepted, with a non-sensitive explanation.
4. The current session must proceed without prior comparison data if no valid prior report is available.

---

## 8. Failure Handling

### 8.1 Valid failure reasons to surface to users

The following failure reasons are safe to communicate to users:
- "This report cannot be verified as a system-generated document."
- "This report is not eligible for comparison."
- "The report file could not be processed."
- "This report was generated with an older format. Please contact support."

### 8.2 Failure reasons that must not be surfaced to users

The following must never be shown to users:
- internal error codes (e.g. `INVALID_VCODE`)
- technical field names
- verification logic details
- secret-related errors

All sensitive failure detail must be logged internally only.

---

Comparison eligibility may be further restricted by session type.

| Session Type | Comparison Supported |
|---|---|

---

## 10. Data Retention and Comparison Window

The comparison window (how old a prior report can be and still be used for comparison) is a product decision.

Current default: **no time-based restriction** unless product explicitly sets one.

Future versions of this policy may define maximum report age for comparison eligibility.

---

## 11. Test and Demo Environment Policy

Reports generated in non-production environments (test, staging, demo) must:
- have `Comparison Eligibility: NO` by default,
- OR be explicitly excluded by the backend before comparison is offered to the user.

Test reports must not be accepted as valid prior reports in production comparison flows.

---

## 12. Audit and Logging Requirements

Every comparison verification attempt must be logged with:
- timestamp
- report ID (if extracted)
- verification result status
- failure reason (if applicable)
- session type
- schema version

Logs must not contain:
- secrets
- raw VCODE values
- full report content
- user conversation data

---

## 13. Support Policy

Support teams may:
- receive the verification status and failure reason category
- relay non-sensitive failure explanations to users

Support teams must not:
- manually override verification results
- mark a failed report as valid
- attempt to reconstruct or re-sign reports

All re-verification must go through the system.

---

## 14. Policy Versioning

This policy must be versioned when any of the following change:
- eligibility criteria,
- session type support list,
- failure handling rules,
- comparison window rules.

Changes to this policy require explicit engineering and product sign-off.
